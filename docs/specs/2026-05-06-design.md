# Claude Code History Manager - Design Spec

**Date:** 2026-05-06
**Status:** Approved (brainstorm complete, awaiting implementation plan)
**Tool name:** `chist` (claude history)

## Problem

Claude Code already persists every session as JSONL under `~/.claude/projects/<sanitized-cwd>/<session-id>.jsonl`. As of design time, the user has 15 projects, 255 sessions, and 153 MB of transcripts on disk.

The raw history exists but is not usable:
- No way to search across sessions ("find that chat about foreign catalogs").
- No way to resume an old session's context in a new session.
- No way to export a session to a readable format.
- No way to surface recent context when starting a fresh session in a project.

## Goals

A single tool, `chist`, that provides:

1. **Search** across all past sessions with full-text ranking and filters (project, date, role).
2. **Resume** an old session by feeding its content (summary or full) into the current Claude Code session.
3. **Export** a session to Markdown for archival or sharing.
4. **Auto-distill** key decisions, files touched, and commands run from a session into a brief summary.
5. **Project memory** via a session-start banner showing the most recent session in the current project.

## Non-Goals

- Cross-machine sync (single-machine tool).
- LLM-quality summaries (heuristic only in v1; LLM summary is a v2 `--llm` flag).
- Tagging or favoriting sessions (out of scope for v1).
- Web UI (CLI plus Claude Code integration only).
- Replacing Claude Code's native session persistence (we read from it, never write).

## Architecture Overview

```
~/.claude/projects/<project>/<session-id>.jsonl   # source of truth (Claude Code writes these)
        |
        | SessionEnd hook -> chist index --incremental
        v
~/.claude/history.db                              # SQLite FTS5 index (built/maintained by chist)
        |
        +-- chist search / list / show           (terminal CLI)
        +-- chist resume / export                (terminal CLI)
        +-- chist banner                         (SessionStart hook)
        |
        +-- /history-search, /history-resume, /history-export   (slash commands shelling to chist)
        +-- claude-history skill                                (auto-trigger natural-language flow)
```

Three layers:

1. **CLI** (`~/tools/claude-history/`, Python 3 stdlib only) does all the work.
2. **Slash commands** (`~/.claude/commands/`) shell out to the CLI from inside Claude.
3. **Skill** (`~/.claude/skills/claude-history/`) auto-triggers the CLI on natural-language requests.

Hooks in `~/.claude/settings.json` keep the index fresh (`SessionEnd`) and surface prior context (`SessionStart`).

## Data Model

### SQLite Schema (`~/.claude/history.db`)

```sql
-- one row per JSONL file
CREATE TABLE sessions (
    session_id   TEXT PRIMARY KEY,           -- UUID from filename
    project      TEXT NOT NULL,              -- sanitized cwd, e.g. "-workspace-databricks-account-infra"
    file_path    TEXT NOT NULL,              -- absolute path to JSONL
    file_mtime   REAL NOT NULL,              -- last seen mtime (epoch); used for incremental indexing
    started_at   TEXT,                       -- ISO timestamp of first message
    ended_at     TEXT,                       -- ISO timestamp of last message
    msg_count    INTEGER NOT NULL DEFAULT 0,
    archived     INTEGER NOT NULL DEFAULT 0  -- 0 = live JSONL, 1 = gzipped in ~/.claude/archive/
);

CREATE INDEX idx_sessions_project ON sessions(project);
CREATE INDEX idx_sessions_started ON sessions(started_at);

-- one row per message (user/assistant/tool_use/tool_result)
CREATE TABLE messages (
    msg_id       INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id   TEXT NOT NULL REFERENCES sessions(session_id) ON DELETE CASCADE,
    seq          INTEGER NOT NULL,           -- 0-based position within session
    role         TEXT NOT NULL,              -- user | assistant | tool_use | tool_result
    timestamp    TEXT,                       -- ISO timestamp from JSONL if present
    content      TEXT NOT NULL,              -- flattened message text (json-stringified for tool calls)
    UNIQUE(session_id, seq)
);

CREATE INDEX idx_messages_session ON messages(session_id);
CREATE INDEX idx_messages_role ON messages(role);

-- FTS5 index over message content
CREATE VIRTUAL TABLE messages_fts USING fts5(
    content,
    role UNINDEXED,
    session_id UNINDEXED,
    seq UNINDEXED,
    content='messages',
    content_rowid='msg_id',
    tokenize='porter unicode61'
);

-- triggers keep FTS in sync with messages
CREATE TRIGGER messages_ai AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content, role, session_id, seq)
    VALUES (new.msg_id, new.content, new.role, new.session_id, new.seq);
END;
CREATE TRIGGER messages_ad AFTER DELETE ON messages BEGIN
    INSERT INTO messages_fts(messages_fts, rowid, content, role, session_id, seq)
    VALUES ('delete', old.msg_id, old.content, old.role, old.session_id, old.seq);
END;

-- index metadata (singleton)
CREATE TABLE index_meta (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- expected keys: schema_version, last_full_index_at, last_incremental_at
```

### JSONL Source Format

Each line in a Claude Code session JSONL is a JSON object with shape (observed):

```json
{
  "type": "user" | "assistant" | "tool_use" | "tool_result",
  "message": { "role": "...", "content": [...] },
  "timestamp": "2026-05-06T10:56:00.000Z",
  "uuid": "...",
  ...
}
```

`parser.py` normalizes each line into `{role, timestamp, content}` where `content` is a flattened string (concatenated text blocks; tool calls are JSON-stringified). Schema variations across Claude Code versions are tolerated by the parser, which logs a warning and skips lines it cannot parse rather than failing the whole file.

## CLI Surface

### `chist index`

```
chist index                        # full rebuild: drop and re-populate from all JSONL
chist index --incremental          # only files with file_mtime > sessions.file_mtime
chist index --quiet                # suppress normal stdout (errors still go to stderr)
chist index --stats                # print: project count, session count, message count, db size, archive size, last index times
```

Behavior:
- Walks `~/.claude/projects/*/*.jsonl`.
- For each file, compares `mtime` against `sessions.file_mtime`. If newer (or session row absent), re-parses and upserts messages.
- Upsert strategy: delete existing messages for that session_id, re-insert. Simple and correct; sessions are typically <2 MB, so re-parse cost is negligible.
- Wraps each session's upsert in a `BEGIN IMMEDIATE` transaction. If SQLite is locked, retries up to 5 seconds with exponential backoff, then exits 1.
- Updates `index_meta.last_incremental_at` on success.

### `chist search`

```
chist search "<query>"                                  # FTS5 ranked across all projects
chist search "<query>" --project <name>                 # filter by project (matched against sessions.project; accepts substring)
chist search "<query>" --since 2026-04-01 --until 2026-05-01
chist search "<query>" --role user|assistant|tool_use|tool_result
chist search "<query>" --limit 20
chist search "<query>" --format human|json             # human (default) or machine-readable
```

Output (human format):

```
0060955f  databricks-account-infra  2026-05-04 10:56  ...build or searching for a solution to save the **history** **chat**...
105905dd  databricks-account-infra  2026-04-16 07:24  ...the **history** of catalog migrations needed for...
```

Match terms wrapped in `**` for terminal highlighting (or stripped in `--format json`).

Stale-index check: if any source JSONL has `mtime > index_meta.last_incremental_at`, prints a single-line warning to stderr: `[chist] index stale by N files; run 'chist index --incremental'`. Does not auto-rebuild.

### `chist list`

```
chist list                                             # all sessions, newest first
chist list --project <name>
chist list --since 2026-04-01
chist list --limit 50
chist list --format human|json
```

### `chist show`

```
chist show <session-id-or-prefix>                      # metadata: project, started, ended, msg count, files touched, commands run
chist show <id> --tail 20                              # last N messages
chist show <id> --head 20                              # first N messages
```

Session ID prefix matching: any prefix that uniquely identifies one session works (git-style). Ambiguous prefix exits 1 with the candidate list.

### `chist export`

```
chist export <id>                                      # Markdown to stdout
chist export <id> -o <path>                            # write to file
chist export <id> --since-message N                    # partial export from sequence N onward
```

Markdown shape:

```markdown
# Session <id> - <date> (<project>)

**Started:** 2026-05-06 10:56
**Ended:** 2026-05-06 12:34
**Messages:** 47

---

## User (10:56:01)

<content>

## Assistant (10:56:08)

<content>

### Tool: Bash

```
ls ~/.claude
```

Output:
```
<truncated to 2000 chars>
```
```

### `chist resume`

```
chist resume <id>                                      # default: distilled summary (heuristic)
chist resume <id> --full                               # full transcript
chist resume --last                                    # most recent session for cwd's project
chist resume --last --project <name>                   # most recent session in given project
```

Distilled summary content (from `distill.py`):

```markdown
# Session <id> - <date> (<project>)

## Open threads
(last 5 user messages without an assistant response containing "done", "completed", "fixed", or matching ack heuristics)

## Decisions
(assistant lines with phrases: "Decision:", "Picked", "Going with", or assistant lines following a user "ok"/"yes"/"go"/"approved")

## Files touched
(deduplicated paths from Read/Edit/Write/NotebookEdit tool_use blocks, with operation counts)

## Commands run
(deduplicated commands from Bash tool_use blocks, with run counts and last exit status if recoverable)

## Last assistant message
(verbatim, truncated to 500 chars)
```

Heuristics are intentionally simple and deterministic. No LLM calls. Misses are acceptable; the user can fall back to `--full`.

### `chist banner`

```
chist banner --cwd-project                             # used by SessionStart hook
chist banner --project <name>
```

Single-query SQL: most recent session for the project. Output (or empty if none):

```
[claude-history] Most recent session in this project: 0060955f (2 hours ago, 47 messages)
                 Resume with: /history-resume 0060955f
```

### `chist prune`

```
chist prune --older-than 180d                          # gzip JSONL into ~/.claude/archive/<year>/<project>/, mark archived=1
chist prune --older-than 180d --dry-run                # preview only
```

Archived sessions remain searchable; `chist resume` and `chist export` gunzip on the fly into a tempfile.

### `chist vacuum`

```
chist vacuum                                           # SQLite VACUUM + rebuild FTS5 (defragment)
```

## Slash Commands

Three Markdown files in `~/.claude/commands/`:

### `history-search.md`

```markdown
---
description: Search past Claude Code sessions
---

Run: `chist search "$ARGUMENTS" --project "$(basename "$(pwd)")" --limit 10 --format human`

Show the user the top hits as a numbered list. If they pick one, suggest `/history-resume <id>`.
```

### `history-resume.md`

```markdown
---
description: Resume context from a past session by id (or prefix, or --last)
---

Run: `chist resume $ARGUMENTS`

The output is a Markdown summary of the past session. Read it into context and proceed with the current task informed by it.
```

### `history-export.md`

```markdown
---
description: Export a past session to Markdown
---

Run: `mkdir -p docs/history && chist export $ARGUMENTS -o "docs/history/$(date +%Y-%m-%d)-$(echo "$ARGUMENTS" | head -c 8).md"`

Confirm the path written.
```

## Skill

`~/.claude/skills/claude-history/SKILL.md`:

```markdown
---
name: claude-history
description: Use when the user asks to find, recall, resume, or summarize a past Claude Code conversation. Triggers on phrases like "find that chat about X", "what did we decide about Y last week", "resume the session where we did Z", "show prior sessions on this project", "summarize last week's work". Auto-invokes the chist CLI.
---

# Claude History Skill

When the user asks to find or recall a past conversation:

1. Identify the search query and any filters (project, time range).
2. Default to current project: `chist search "<query>" --project "$(basename "$(pwd)")" --limit 10 --format json`.
3. Parse the JSON, present a numbered list with date, project, and a snippet of each hit.
4. Ask the user which session to resume or export.
5. On confirmation, run `chist resume <id>` for context, or `chist export <id>` for a saved file.
6. If the heuristic summary is insufficient, offer `chist resume <id> --full`.

Do NOT run `chist index` proactively; the SessionEnd hook keeps the index fresh.
```

## Hook Integration

Added to `~/.claude/settings.json`:

```json
{
  "hooks": {
    "SessionEnd": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "chist index --incremental --quiet >> ~/.claude/history-index.log 2>&1 &"
          }
        ]
      }
    ],
    "SessionStart": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "chist banner --cwd-project"
          }
        ]
      }
    ]
  }
}
```

Notes:
- `SessionEnd` is backgrounded (`&`) so session teardown is not blocked.
- `SessionStart` is synchronous but fast (single SQL query); empty output if no prior session.
- Log rotates at 1 MB, keeping the latest 3 (`chistlib/banner.py` and `indexer.py` self-rotate before writing).

## Failure Modes

| Scenario | Handling |
|----------|----------|
| Two sessions end concurrently | SQLite `BEGIN IMMEDIATE` plus 5s backoff retry; second invocation no-ops if first already covered the same files. |
| JSONL written mid-session (file open) | Parser tolerates trailing partial line; mtime-driven incremental will re-index next time. |
| Hook didn't fire (crash, kill -9) | `chist search` checks for stale index and prints a one-line warning. Manual `chist index --incremental` recovers. |
| JSONL schema change in future Claude Code version | Parser logs warnings, skips unparseable lines, never fails the whole file. Schema additions are forward-compatible (extra keys ignored). |
| Disk full during index | Transaction rolls back; existing index remains valid; error logged to `~/.claude/history-index.log`. |
| Archive directory missing on resume of archived session | `chist resume` errors with the expected path; user runs `chist prune --rehydrate <id>` (added if needed) or restores from backup. |
| `chist` not on PATH when hook runs | Hooks fail silently (output to log); banner is empty. Install instructions in README cover symlinking onto PATH. |

## Testing Strategy

- **`tests/fixtures/sample-session.jsonl`**: synthetic 20-message session covering user, assistant, tool_use, tool_result roles plus edge cases (empty content, malformed line, unicode).
- **`test_parser.py`**: round-trip parse, malformed-line tolerance, unicode handling.
- **`test_indexer.py`**: full index, incremental index, idempotency (running twice = same DB), concurrency (two processes upserting different sessions).
- **`test_search.py`**: FTS5 ranking, filters (project, date, role), stale-index warning.
- **`test_distill.py`**: decisions extraction, files-touched extraction, commands extraction, on a curated fixture.
- All tests use a temp `history.db` and temp project directory; no dependency on real `~/.claude/`.
- Run via `python -m unittest discover tests/`.

## Open Decisions Deferred to v2

1. **LLM-quality summaries** (`chist resume <id> --llm`) shelling out to `claude -p`.
2. **Tagging / favoriting sessions** (`chist tag <id> <label>`).
3. **Cross-machine sync** (encrypted backup of `history.db` plus archive to S3).
4. **Web UI** for browsing.
5. **Token-cost estimation** (per-session cost using model pricing tables).

## Acceptance Criteria

For v1 to be considered complete:

1. `chist index` indexes all 255 existing sessions in under 30 seconds on first run.
2. `chist index --incremental` completes in under 2 seconds when no files changed.
3. `chist search "foreign catalog"` returns ranked results in under 200 ms.
4. `chist resume <id>` produces a distilled summary in under 500 ms for sessions up to 5 MB.
5. `chist banner` returns in under 100 ms.
6. `SessionEnd` hook does not delay session teardown (backgrounded).
7. `SessionStart` hook displays a banner without blocking the prompt for more than 100 ms.
8. All slash commands work end-to-end inside a Claude Code session.
9. The skill auto-triggers on at least 5 sample natural-language queries (validated manually).
10. All unit tests pass.

## Decision Log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Standalone repo `~/tools/claude-history/` | Tool is cross-project; coupling to any single workspace would limit reuse. |
| 2 | Python 3 stdlib only | `sqlite3`, `json`, `argparse` cover everything; no `pip install` required. |
| 3 | SQLite FTS5 over ripgrep | Ranked results plus filters (project, date, role) earn their keep at 255+ sessions; single-file DB, no daemon. |
| 4 | Mtime-keyed incremental indexing | Simpler than file watching; <2s for typical sessions; correct under concurrent writes thanks to upsert-by-session. |
| 5 | Heuristic distill (no LLM) | Deterministic, free, fast; LLM summaries deferred to v2 behind `--llm`. |
| 6 | Hybrid surface (CLI + slash commands + skill) | CLI for terminal use, slash commands for explicit Claude invocation, skill for natural-language triggers. |
| 7 | `SessionEnd` plus `SessionStart` hooks | Auto-index keeps search fresh; banner surfaces continuity for free. |
| 8 | Archive is manual and gzipped, kept indexed | User controls when to prune; archived sessions remain searchable. |
